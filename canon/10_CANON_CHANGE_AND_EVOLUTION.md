# 10_CANON_CHANGE_AND_EVOLUTION

**Status:** CANONICAL  
**Version:** v0.1

## Purpose
Defines the mechanism for updating these governing documents, ensuring stability while allowing necessary evolution.

## Hard Constraints
*   **Amendment Process:** A change to Canon requires a specific "Canon Amendment Proposal" (CAP) artifact.
*   **High Bar:** Canon should only change if a constraint is proven unworkable or existential to the product's failure.
*   **Versioning:** All changes increment the Canon version (v0.2, v1.0). Old rules remain effectively binding until explicitly superseded.
*   **Backward Compatibility:** New rules must specify how to handle legacy data/features that violate them (e.g., "Deprecate" or "Migrate").

## Explicit Non-Goals
*   **No "Silent Updates":** Canon cannot be changed via casual conversation or code comments.
*   **No Forking:** There is only one Canon.
